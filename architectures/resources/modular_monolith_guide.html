§ 01

Building Intuition

Before touching definitions, let's build the right mental picture — because most engineers misunderstand what a modular monolith actually is.

🏢 The Office Building Analogy

Imagine a chaotic open-plan office. Everyone sits together: the sales team, engineers, accountants, HR. They constantly interrupt each other, share the same coffee machine, shout across the floor. Decisions affect everyone. This is a spaghetti monolith — everything tangled with everything.

Now imagine the same building, but with clearly partitioned departments. Sales has their floor, Engineering has theirs, Finance has its own wing. Each department has its own processes, its own filing system, and deals with others only through formal inter-department requests. They still share the building's electricity, elevators, and reception. That's a modular monolith.

The key insight: it's still one building (one deployable application), but internally it's organized into strict, bounded zones. Order through structure — not by blowing up the building and renting 50 separate offices (microservices).

The Spectrum

ANTI-PATTERN
Spaghetti Monolith
One codebase, zero structure. Classes call each other freely. No concept of ownership. The classic "big ball of mud." Every change is dangerous.
THE MIDDLE PATH →
Modular Monolith
One deployable app, strict internal module boundaries, clear ownership, controlled interfaces. The discipline of microservices applied inside a single process.
COMPLEX ALTERNATIVE
Microservices
Many independent processes. Full operational freedom, but enormous infrastructure overhead. Only justified at significant team and traffic scale.

The Core Mental Model

A modular monolith answers a very specific question: "How do I get the team-independence and clean boundaries of microservices, without the network complexity, distributed system problems, and operational overhead?"

The answer: enforce boundaries in code, not in infrastructure. Instead of making services talk over HTTP, make modules talk through well-defined Java interfaces. You get 80% of the benefits at 20% of the cost.

💡
The Architect's Secret
Most successful microservice migrations started as well-structured modular monoliths. If your module boundaries are clean in a monolith, extracting a module into a microservice later becomes a relatively mechanical task — swap the interface call for an HTTP call. Start modular, go distributed only when pain demands it.
§ 02

What Is Modular Monolith?

A modular monolith is a single deployable application that is internally organized into strongly encapsulated, loosely coupled modules — each owning a distinct business domain.

There are two parts to the name and both matter equally:

PART 01
Monolith
It's one application. One build artifact (e.g., a single JAR). Deployed as one unit. Runs in one process. Services communicate via method calls — not network requests. Simple to develop, test, and operate.
PART 02
Modular
Internally divided into strict modules. Each module has a public API (interface), private internals, and owns its own domain. Cross-module access is controlled and explicit. No tangled dependencies.

The Three Inviolable Rules

// RULE 1: Modules communicate ONLY through their public API // ✓ Correct — calling the public interface OrderService orderService = context.getBean(OrderService.class); orderService.placeOrder(orderRequest); // ✗ VIOLATION — reaching into another module's internals OrderRepositoryImpl repo = new OrderRepositoryImpl(); // BANNED // RULE 2: No circular dependencies between modules // Order → Payment ✓ | Order → Payment → Order ✗ // RULE 3: Enforce boundaries with package-level visibility // Internal classes are package-private, not public class OrderRepositoryImpl { ... } // package-private = hidden public interface OrderService { ... } // public = module API

How It Compares

Dimension Spaghetti Monolith Modular Monolith Microservices
Deployment Single JAR Single JAR Many independent services
Internal Structure None — free-for-all Strict module boundaries Each service is isolated by process
Communication Direct method calls, anywhere Method calls via public APIs only HTTP / gRPC / messaging over network
Data One shared DB, all tables shared One DB, schemas per module (or logical separation) Separate DB per service
Scalability Scale whole app only Scale whole app only Scale individual services
Operational Complexity Low Low High
Migration Path Hard to migrate — tangled Clean migration path to microservices Already distributed
§ 03

Problems It Solves

A modular monolith targets a very specific set of problems — those caused by the unstructured monolith, without introducing the distributed-systems pain of microservices.

The Big Ball of Mud Problem

The most common failure mode of growing software is the "Big Ball of Mud" — a monolith where any class can reference any other class. Over time, you end up with 500 classes that all depend on each other in unpredictable ways. Change anything, and you don't know what breaks. No one understands the full system.

Spaghetti Monolith Problem Real Consequence Modular Monolith Solution
🍝 Tangled Dependencies Changing the Order model requires touching 40 unrelated files Module boundary enforces that Order's internals are private. Only the public API is visible.
💥 Accidental Coupling Developer A's shortcut creates a dependency from Billing into User's internal cache Compiler/access modifiers prevent this. Package-private classes are simply invisible to other modules.
🧪 Untestable Code Can't unit test Order logic without spinning up Payments, Users, and Notifications Each module is independently testable via its public interface. Other modules are mocked.
👥 Team Collision 5 devs editing the same files daily — constant merge conflicts, broken builds Each team owns a module. Clear file ownership means minimal conflicts.
🔍 No Cognitive Boundary New engineer must understand the whole system to change one feature A new engineer on the Payments team only needs to understand the Payments module.
🚧 Fear of Refactoring "I'm afraid to change this class — it's referenced everywhere" Module internals can be refactored freely. Only the public API contract must be preserved.

Why Not Just Use Microservices?

PROBLEM 01
Too Small a Team
A team of 8 running 10 microservices means everyone is on-call for multiple services, managing 10 CI pipelines, 10 Docker images. The overhead kills velocity.
PROBLEM 02
Domain Uncertainty
You don't know the right service boundaries yet. Microservices with wrong boundaries are worse than a monolith — you get tight coupling across a network.
PROBLEM 03
No DevOps Maturity
No Kubernetes expertise, no service mesh, no distributed tracing set up. Microservices without this infrastructure become a support nightmare.
PROBLEM 04
Distributed Complexity
Network failures, eventual consistency, distributed transactions — all brand new problems. Most teams aren't ready to handle these on top of feature development.
💡 The Core Value Proposition

A modular monolith gives you clean architecture, team autonomy, and easy testability — all the "developer experience" wins of microservices — while keeping simple deployment, in-process calls, and ACID transactions — all the operational wins of a monolith. It's not a compromise; for many systems, it's the optimal architecture.

§ 04

Key Components

A modular monolith has fewer infrastructure components than microservices, but requires careful attention to its internal architecture. Here's every piece.

📦
Business Modules (Domain Modules)
The heart of the architecture. Each module encapsulates one business domain — Users, Orders, Payments, Catalog, Notifications. Each has a public API (interface), internal implementation classes, and its own data access layer. The golden rule: no module ever imports the internal classes of another module.
e.g. order-module, payment-module, user-module
🔌
Public Module API (Facade / Interface Layer)
Every module exposes a narrow, intentional public interface — typically a set of Java interfaces and DTOs (Data Transfer Objects). This is the only surface area another module is allowed to interact with. The implementation behind the interface is completely hidden. This is what enforces the boundary.
e.g. OrderService.java, OrderRequest.java, OrderResult.java
🌐
API / Web Layer (Shared Entry Point)
The application's entry point — typically REST controllers in Spring Boot. This layer is thin: it translates HTTP requests into calls to the appropriate module's public API. It doesn't contain business logic. Think of it as the receptionist who routes requests to the right department.
Spring MVC Controllers, request/response mappers
🗄️
Database with Schema-per-Module
One physical database, but each module "owns" its set of tables — enforced by convention (table prefix, separate schema) or tooling (ArchUnit tests). Module A's repository never queries Module B's tables directly. If cross-module data is needed, it goes through the public API. This mirrors the microservice database isolation principle, at lower cost.
PostgreSQL with schemas: orders.*, users.*, payments.*
📡
Internal Event Bus (Optional)
For decoupled cross-module communication that avoids direct method calls. Module A publishes an in-process event ("OrderPlaced"), Module B listens without knowing about Module A. In Spring Boot, this is Spring's ApplicationEventPublisher. Enables the same decoupling pattern as Kafka, but in-memory and without the infrastructure.
Spring ApplicationEventPublisher, Guava EventBus
🔒
Shared Kernel (Carefully)
A small, stable shared library used by all modules — common value objects (Money, Address, UserId), shared exceptions, utility classes, and base domain types. Must be kept minimal and stable. If it changes constantly, every module breaks. The rule: if you're unsure whether something belongs here, it probably doesn't.
common-domain: Money.java, UserId.java, DomainException.java
🏗️
Dependency Enforcement Tooling
The boundary rules are meaningless unless enforced automatically. Architecture tests (ArchUnit for Java) write rules like "no class in package X may access class in package Y" and run them in CI. These tests catch boundary violations before they merge, keeping the modular structure honest over time.
ArchUnit, JDepend, Maven Enforcer Plugin
🚀
Single Deployment Unit
All modules are compiled and bundled into one artifact — a fat JAR in Spring Boot. One CI/CD pipeline, one Docker image, one Kubernetes deployment. No inter-service networking to configure. Deploy the whole app; each module is just code inside it. This dramatically simplifies operations compared to microservices.
One Spring Boot fat JAR, one Docker image, one deployment
§ 05

How It Is Applied

Applying a modular monolith is fundamentally a question of how you divide your system into modules and how you enforce those boundaries in code.

Step 1 — Identify Your Modules (Bounded Contexts)

Use Domain-Driven Design to find your natural module boundaries. Look for areas of the business that have their own language, their own concepts, and their own lifecycle. Each bounded context becomes a module.

// E-commerce example — module identification com.myapp.users // User accounts, authentication, profiles com.myapp.catalog // Products, categories, pricing, inventory com.myapp.orders // Order lifecycle, cart, checkout flow com.myapp.payments // Payment processing, refunds, invoices com.myapp.shipping // Fulfillment, tracking, warehousing com.myapp.notifications // Email, SMS, push notifications com.myapp.common // Shared kernel — Money, UserId, etc.

Step 2 — Structure Each Module Consistently

Every module follows the same internal structure. This predictability is what makes the codebase easy to navigate, even as it grows.

// Internal structure of the 'orders' module com.myapp.orders/ ├── api/ // PUBLIC — visible to other modules │ ├── OrderService.java // interface (the contract) │ ├── OrderRequest.java // input DTO │ └── OrderResult.java // output DTO │ ├── domain/ // PRIVATE — internal business logic │ ├── Order.java // domain entity (package-private) │ ├── OrderItem.java // value object (package-private) │ └── OrderStatus.java // enum (package-private) │ ├── application/ // PRIVATE — use case orchestration │ └── OrderServiceImpl.java // implements OrderService │ ├── infrastructure/ // PRIVATE — DB, external calls │ ├── OrderRepository.java │ └── OrderJpaEntity.java │ └── events/ // PRIVATE — events this module publishes └── OrderPlacedEvent.java

Step 3 — Enforce Boundaries with ArchUnit

// ArchUnit test — boundaries enforced in CI @AnalyzeClasses(packages = "com.myapp") class ModuleBoundaryTest { @ArchTest static final ArchRule ordersCannotAccessPaymentInternals = noClasses() .that().resideInAPackage("..orders..") .should().accessClassesThat() .resideInAPackage("..payments.domain.."); // This test FAILS the build if a developer breaks the boundary }

Step 4 — Define Cross-Module Communication

PATTERN A
Direct API Call
When Module A needs data from Module B immediately. Inject Module B's PublicService interface into Module A and call it. Clean, simple, type-safe. Use for synchronous read operations.
PATTERN B
Domain Event
When an action in Module A should trigger reactions in multiple modules without coupling them. Publish an in-process event. Other modules subscribe independently. Use for notifications, side effects.
§ 06

How It Works

Under the hood, a modular monolith runs in a single JVM process. All modules share memory, the thread pool, and the Spring application context — but are disciplined about what they expose.

Module Loading and Wiring (Spring Boot)

All modules are scanned by Spring on startup. Each module can register its own beans, configuration, and repositories. Spring acts as the "dependency injection glue" that wires them together — but the interfaces ensure that modules only see what they're meant to see.

// Each module has its own Spring @Configuration class @Configuration @ComponentScan("com.myapp.orders") // only scans this module's classes @EnableJpaRepositories(basePackages = "com.myapp.orders.infrastructure") public class OrdersModuleConfig { } // The main app just imports all module configs @SpringBootApplication @Import({ OrdersModuleConfig.class, PaymentsModuleConfig.class, UsersModuleConfig.class, CatalogModuleConfig.class }) public class Application { }

Synchronous Inter-Module Call

When Order module needs to check the product price from Catalog module, it calls through the public interface. This is a plain Java method call — no network, no serialization, nanosecond latency, ACID-safe.

// OrderServiceImpl.java — inside the Order module public class OrderServiceImpl implements OrderService { // Injecting the PUBLIC interface of CatalogModule — not its impl! private final CatalogService catalogService; // interface from catalog.api private final PaymentService paymentService; // interface from payments.api public OrderResult placeOrder(OrderRequest req) { Price price = catalogService.getPrice(req.productId()); // in-process call PaymentResult payment = paymentService.charge(req.userId(), price); // in-process Order order = Order.create(req, price, payment.transactionId()); orderRepository.save(order); eventPublisher.publishEvent(new OrderPlacedEvent(order.id())); return new OrderResult(order.id(), OrderStatus.CONFIRMED); } }

Asynchronous Event (In-Process)

After the order is saved, an event is published via Spring's event bus. Notification and Inventory modules react independently — they don't need to be called explicitly. Adding a new reactor requires no change to the Order module.

// NotificationsModule — listens for events from any module @Component public class OrderNotificationHandler { @EventListener // no dependency on Order module's internals! public void on(OrderPlacedEvent event) { emailService.sendConfirmation(event.orderId()); } } // If you need the handler to run in a different thread: @Async // truly non-blocking — runs on Spring's TaskExecutor thread pool @TransactionalEventListener(phase = AFTER_COMMIT) public void on(OrderPlacedEvent event) { ... }

Database: Schema-per-Module

One PostgreSQL instance, multiple schemas — each owned by a module. JPA entities in the orders module only reference tables in the orders schema. Cross-schema JOINs in repositories are prohibited (enforced via ArchUnit or code review).

-- PostgreSQL schemas per module CREATE SCHEMA orders; -- owned by OrderModule CREATE SCHEMA payments; -- owned by PaymentsModule CREATE SCHEMA users; -- owned by UsersModule -- OrderModule can query freely within its own schema SELECT * FROM orders.orders WHERE status = 'PLACED'; -- ✗ BANNED — cross-schema JPA query by OrderModule SELECT o.*, u.email FROM orders.orders o JOIN users.users u ON o.user_id = u.id; -- cross-module JOIN = violation

§ 08

Architectural Request Flow

Let's trace a complete "Place Order" request end to end — every hop, every module interaction, every database write.

1
HTTP Request → REST Controller
POST /api/orders arrives at the Spring MVC dispatcher. The OrderController (in the API layer) validates the request body, extracts the authenticated user ID from the JWT, and maps the HTTP payload into an OrderRequest DTO. No business logic here — just HTTP-to-Java translation.
2
Controller → OrderService (Module API)
The controller calls orderService.placeOrder(request) — the Order module's public interface. Spring injects the real implementation (OrderServiceImpl). We're now inside the Order module. Everything from here is in-process Java — no network, no serialization overhead.
3
Order Module → Catalog Module (price lookup, sync)
OrderServiceImpl needs the current product price. It calls catalogService.getPrice(productId) — the Catalog module's public API. This is a direct Java method call (nanosecond latency). The Catalog module's internals — JPA entity, repository — are completely invisible to Order.
4
Order Module → Payment Module (charge, sync)
paymentService.charge(userId, price) is called. Payment module processes the charge (calls the payment gateway, persists a transaction record in its own payments.* schema). Returns a PaymentResult with the transaction ID. This is still in-process — wrapped in the same database transaction if desired.
5
Order persisted + Event published (within @Transactional)
Order module persists the Order entity in orders.orders table. Then publishes an OrderPlacedEvent via Spring's ApplicationEventPublisher. Using @TransactionalEventListener(phase = AFTER_COMMIT) ensures the event only fires after the database transaction commits successfully — no phantom events on rollback.
6
HTTP Response → Client
OrderService returns an OrderResult. The controller maps it to a JSON response: 201 Created with the order ID. Typically sub-50ms end-to-end, since all calls are in-process. No network hops between modules.
7
Event Listeners fire asynchronously (after commit)
After the transaction commits, Spring fires the OrderPlacedEvent to all listeners on a background thread pool. NotificationModule: sends confirmation email. ShippingModule: creates a shipment record. AnalyticsModule: records the sale. None of these know about each other — pure event-driven decoupling, all in-process.

The Transaction Advantage

💎
ACID Transactions Across Modules — For Free
Because all modules share one database connection pool, steps 2–5 can be wrapped in a single @Transactional boundary. If payment fails, the order is NOT saved. If order save fails, the payment record is rolled back. This is trivially free in a modular monolith. In microservices, this requires a Saga pattern — complex compensating transactions, eventual consistency, and significant engineering effort. This is one of the most underrated advantages of the modular monolith.
§ 09

Use Cases & Skip Cases

A modular monolith isn't for everyone and every stage. Here's the honest assessment of when it's the right call.

✅ Use Modular Monolith When

  • Team has 5–30 engineers — big enough for structure, small enough that microservice overhead kills velocity
  • Domain boundaries are still being discovered — avoid locking in wrong service boundaries early
  • You need ACID transactions across multiple business domains (orders + payments, etc.)
  • Low to medium traffic — no need to scale individual features independently
  • Limited DevOps maturity — no Kubernetes expertise yet, no need for it
  • Building a product from scratch — establish clean architecture before premature distribution
  • B2B SaaS, internal tools, enterprise apps with complex domain logic
  • You want a clear migration path to microservices when the time is right
  • Your current monolith is a "big ball of mud" — modularize before extracting

🚫 Skip It When

  • Individual features have radically different scaling requirements that demand independent scaling right now
  • Teams are 50+ engineers — coordination overhead demands true service independence
  • Different features require different technology stacks that are genuinely incompatible
  • You have features with vastly different availability SLAs (99.9% core vs 95% analytics)
  • You're already running microservices well — going backwards rarely makes sense
  • Features must be independently deployable by separate release trains
  • Strong regulatory requirements mandate strict physical data isolation between domains

The Goldilocks Zone

Company Stage Engineers Recommended Architecture Why
🌱 Seed / Pre-product 1–4 Simple Monolith Ship fast, validate the idea. No time for architecture.
📈 Early Growth 5–12 Modular Monolith Add structure before the codebase becomes unmaintainable.
🏢 Growth 12–40 Modular Monolith (primary) Right-sized for team. Modules provide team independence without infra overhead.
🚀 Scale 40–100 Selective Microservices Extract 2–3 modules with proven scaling needs. Keep the rest modular.
🌐 Enterprise 100+ Full Microservices Teams need full autonomy. Organizational scale demands it.
🔑 The Key Insight

Shopify, Stack Overflow, and Basecamp have all run significant portions of their business on well-structured monoliths at substantial scale. The modular monolith is not a stepping stone you're forced to abandon — for many companies, it's the permanent, correct answer. The goal is always to match architecture to organizational reality, not to cargo-cult whatever Netflix does.

§ 10

Pros & Cons

An honest accounting of what you gain and give up — relative to both a spaghetti monolith and microservices.

✅ Advantages

  • Simple Deployment — One JAR. One CI/CD pipeline. One Docker image. Deploy in 2 minutes, not 45.
  • ACID Transactions — Cross-module operations in one database transaction. Free consistency that costs enormous effort in microservices.
  • Zero Network Latency — All inter-module calls are Java method invocations. No serialization, no timeouts, no retries.
  • No Distributed System Problems — No eventual consistency, no network partitions, no distributed tracing needed.
  • Clean Codebase — Enforced boundaries prevent the "big ball of mud." New engineers understand their module without learning the whole system.
  • Easy Refactoring — Module internals can be completely rewritten without breaking other modules, as long as the public API is preserved.
  • Microservices Migration Path — When a module is ready to be extracted, boundaries are already clean. It's a mechanical transformation.
  • Low Operational Cost — No Kubernetes expertise needed. No service mesh, no distributed logging infrastructure required from day one.
  • Easy End-to-End Testing — Spin up one Spring Boot application for integration tests. No service mocking infrastructure.

✗ Limitations

  • Whole-App Scaling Only — Can't scale just the "search" feature independently. Must scale the entire application.
  • Single Technology Stack — All modules must use the same language and (mostly) the same framework. Can't use Python for ML in one module.
  • Shared Deployment Risk — A catastrophic bug in any module can crash the whole application. No fault isolation between modules.
  • Discipline Requirement — Boundaries only hold if the team enforces them. Without ArchUnit tests and code review culture, they erode over time.
  • Shared Thread Pool — One slow module (e.g., a blocking I/O call) can consume all available threads and starve the rest of the application.
  • Database Sharing Risk — One misconfigured slow query can degrade performance for all modules. With microservices, each service has its own pool.
  • Limited Team Autonomy at Scale — At 50+ engineers, even clean module boundaries can feel constraining. True independence eventually requires separate deployables.
  • Long Build Time (at extremes) — At 20+ modules, build and test times grow. Mitigated by parallel builds but eventually a concern.

§ 11

Things to Be Careful Of

These are the patterns that silently corrupt a modular monolith back into a spaghetti ball. Every team that doesn't actively guard against them eventually faces them.

🔥
Boundary Erosion — "Just This Once"
The #1 killer. A developer needs something from another module "just this once" — they make the internal class public instead of exposing it through the proper API. Then it happens again. Within 18 months, the boundaries are meaningless. Counter-measure: ArchUnit tests in CI that fail the build on violations. Automated enforcement, not trust.
💥
Shared Database Tables (The Silent Killer)
Module A writes directly to a table "owned" by Module B. This creates hidden coupling that doesn't show up in code reviews — only at runtime, when Module B's schema change breaks Module A in production. Enforce table ownership strictly: each module's JPA entities may only map to that module's schema. Automated with ArchUnit or naming convention tests.
🔄
Circular Dependencies
Order depends on Payment, Payment depends on Order. Now you can't change either without coordinating with the other — and you can't extract either into a microservice later because they're circular. Rule: dependencies flow in ONE direction. Draw your module dependency graph and make sure it's a DAG (Directed Acyclic Graph). Enforced with ArchUnit or Maven module structure.
⚠️
Anemic Module APIs (Too Much Leakage)
The public API exposes so many methods and raw domain objects that it's essentially making the whole module public. This defeats encapsulation — callers can now do anything with the module's data, not just what the module intends. Keep APIs intentional and minimal. Expose only what external modules genuinely need. Use dedicated DTOs, not your JPA entities.
⚠️
The God Module / God Service
One module grows to own 60% of the system's logic. Everything else depends on it. This is just a modular monolith with bad boundaries. Revisit your domain model. If "Order" is doing pricing logic, user management, and inventory management — those are three modules that got accidentally merged. The module should be cohesive around one business concept.
⚠️
Transactional Event Side-Effects
Using @EventListener (synchronous, same transaction) instead of @TransactionalEventListener(AFTER_COMMIT) means your event handler runs inside the calling transaction. If the handler fails, it can roll back the original transaction unexpectedly. Always use AFTER_COMMIT for event handlers that are meant to be side-effects (notifications, analytics). Understand exactly which transaction boundary your event handler participates in.
⚠️
Thread Pool Starvation
In a shared process, a blocking call in one module (e.g., calling a slow third-party API synchronously) uses a Tomcat thread. If enough requests hit that slow path, all threads are consumed and the whole application becomes unresponsive — even the fast modules. Use @Async for anything slow, or configure separate thread pools for blocking I/O in different modules.
💡
The Architecture Test Suite is Not Optional
In microservices, boundaries are enforced by the network — you literally can't call a private class in another service. In a modular monolith, Java's access modifiers are the only enforcement mechanism — and they can be bypassed with reflection, or simply ignored with public classes. ArchUnit tests are your substitute for the network boundary. They must be written on day one and run on every CI build. Without them, your modular monolith degrades into a spaghetti monolith within 12–18 months.

The Golden Rules

RULE 1: Every module has ONE public interface. Internals are package-private. RULE 2: Modules talk through APIs only. Never import internal classes of another module. RULE 3: Dependencies are acyclic. No circular dependencies, ever. RULE 4: Each module owns its tables. No cross-module JPA joins or direct table access. RULE 5: Expose DTOs, not domain entities, in public APIs. RULE 6: ArchUnit tests enforce boundaries in CI. Automated, not manual. RULE 7: Shared Kernel is minimal. If in doubt, keep it OUT of shared. RULE 8: Use @TransactionalEventListener(AFTER_COMMIT) for event side-effects.
